## HIL Iteration Note (Stage 4)

**The gap I found and fixed.** When I reviewed the LLM's first spec draft, the most consequential gap was in the income-statement specification (Part A, Sections 3 and 6). The draft mapped Zoom's depreciation & amortization to the template's separate `INC_depreciation` row and listed EBIT simply as `INC_sales - INC_cost_goods_sold - INC_sga - INC_depreciation`. I had already caught the underlying problem at Stage 3: Zoom's D&A is *already embedded* inside cost of revenue and operating expenses in its 10-K, so subtracting it again on a separate line double-counts it and understates EBIT by ~$122.6M (it would show ~$690.7M instead of Zoom's reported income from operations of $813.3M).

Left unspecified, a Stage 5 LLM reading only this spec would have reproduced the same double-count and computed wrong EBIT, profitability ratios, and Du Pont operating margin. So I revised the spec in two places: (1) in the Data Inputs table I set `INC_depreciation = 0` with the explicit note "D&A already embedded in opex"; and (2) I added Validation Rule #2 requiring the executor to confirm `INC_ebit` ties to Zoom's reported $813.3M, with the reason stated. I also added Validation Rule #6 documenting the related cash-flow caveat (the simplified tab omits ~$931.3M of stock-based compensation, so its net-change figure intentionally does not tie to the 10-K, and no ratio depends on it). This converts a silent error into an explicit, checkable instruction.
